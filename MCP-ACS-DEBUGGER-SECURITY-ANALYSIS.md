# MCP ACS Debugger — 23-Point Security Analysis

**Tool:** MCP ACS Debugger (packages: `ts-mcp-debugger` VS Code extension, `@ai-capabilities-suite/mcp-debugger-server`, `@ai-capabilities-suite/mcp-debugger-core`)
**Version Analyzed:** 1.5.28 (extension), 1.5.23 (server), 1.0.16 (core)
**Provider:** Digital Defiance (501(c)(3) nonprofit, digitaldefiance.org)
**Repository:** https://github.com/Digital-Defiance/ai-capabilities-suite
**Date:** March 11, 2026

| S.No | Security Check Type | Response |
|------|-------------------|----------|
| 1 | **Licensing & Legal Compliance** | **MIT License.** Fully permissive — allows commercial use, modification, distribution, sublicensing, and private use with no restrictions beyond preserving the copyright notice. Confirmed in the root `LICENSE` file and in each package's `package.json`. No copyleft obligations. Safe for internal and commercial use. |
| 2 | **Data Protection & Privacy Laws** | **Medium risk — mitigations exist but are opt-in.** The debugger inspects runtime variables, call stacks, and evaluates arbitrary expressions, which may expose PII or secrets present in application memory. The core library includes a `DataMasker` module that can detect and redact emails, phone numbers, credit cards, SSNs, API keys, and passwords using regex patterns. However, data masking is **disabled by default** (`enabled: false`). No data is transmitted to external servers — all processing is local (stdio or localhost TCP). No telemetry or analytics collection was found in the source code. **Recommendation:** Enable `DataMasker` when debugging applications that handle sensitive data. |
| 3 | **Infrastructure & Deployment Security** | **Low risk — local-only by default.** The MCP server runs locally via stdio transport (default) or TCP bound to `localhost`. The `process-spawner` binds the Node.js inspector to `127.0.0.1` on a random port (`--inspect-brk=0`). No cloud services, external APIs, or remote endpoints are contacted. The VS Code extension runs within the VS Code sandbox. CI/CD uses GitHub Actions with standard `actions/checkout@v4` and `actions/setup-node@v4`. The server supports graceful shutdown handlers and signal handling. **Recommendation:** If using TCP transport, ensure firewall rules prevent external access. |
| 4 | **Long-Term Viability Risk** | **Medium risk.** The provider (Digital Defiance) is a registered US 501(c)(3) nonprofit with a public website, GitHub organization, and named contributors (Jessica Mulein). The project is actively maintained (v1.5.x with recent npm publishes). It is published on npm, VS Code Marketplace, Docker Hub, and listed on LobeHub MCP registry. However, it is a relatively small nonprofit-backed project, not a major corporation. Dependencies are mainstream (`@modelcontextprotocol/sdk`, `zod`, `ws`, `source-map`). **Risk factors:** Single primary maintainer, nonprofit funding model. |
| 5 | **Unexpected Remote Code Execution and Code Attacks** | **High inherent risk — by design, mitigated by architecture.** The debugger's core function is to spawn and execute Node.js processes (`child_process.spawn`), evaluate arbitrary JavaScript expressions (`Runtime.evaluate` via CDP), and inspect running code. This is inherently a code execution tool. **Mitigations:** (1) It only executes what the AI agent or user explicitly requests — no autonomous execution. (2) The `debugger_start` tool requires explicit `command` and `args` parameters. (3) The `debugger_inspect` tool evaluates expressions only within an already-running debug session context. (4) Resource limits, session timeouts, and circuit breakers constrain runaway execution. **Recommendation:** Never auto-approve `debugger_start` or `debugger_detect_hang` tools. Always review commands before execution. |
| 6 | **Tool Contamination Attacks (Tool Poisoning)** | **Low risk.** The debugger tools are stateless between invocations — each tool call operates on an explicit `sessionId` and returns structured JSON. There is no persistent data store, cache, or shared state that could be poisoned across sessions. The `SessionManager` isolates sessions. Watch expressions and breakpoints are scoped to individual sessions and cleaned up on session stop. No tool reads from or writes to files that could be tampered with to influence future behavior. |
| 7 | **Shadowing Attack** | **Low risk.** The tools are registered with unique, namespaced names (e.g., `debugger_start`, `debugger_set_breakpoint`) via the `@modelcontextprotocol/sdk` `McpServer.registerTool()` API. The VS Code extension registers tools via `languageModelTools` in `package.json` with unique `name` fields. Tool registration happens at server startup in the constructor and is not dynamically modifiable at runtime. There is no mechanism for external code to re-register or override tools after initialization. |
| 8 | **Credential Theft** | **Medium risk.** The debugger can inspect variables and evaluate expressions in running processes, which may contain API keys, tokens, database credentials, or other secrets in memory. The `DataMasker` module (when enabled) redacts common secret patterns. The `AuthManager` uses SHA-256 hashing for API key storage and constant-time comparison to prevent timing attacks. The CLI supports `MCP_AUTH_TOKEN` environment variable. **Risk factors:** (1) `DataMasker` is disabled by default. (2) Variable inspection could expose secrets in AI agent output. (3) Environment variables of spawned processes inherit from the parent. **Recommendation:** Enable data masking; avoid debugging production systems with live credentials. |
| 9 | **Verification of MCP Server Provider** | **Verified — trustworthy.** Digital Defiance is a US-registered 501(c)(3) nonprofit organization (digitaldefiance.org). The GitHub organization (`Digital-Defiance`) is publicly visible with multiple repositories. The primary contributor (Jessica Mulein) is publicly identified with a GitHub profile. The npm packages are published under the `@ai-capabilities-suite` scope. The VS Code extension is published by `DigitalDefiance` on the VS Code Marketplace. The project has a `SECURITY.md` with a responsible disclosure policy (security@digitaldefiance.org) and defined response timelines. |
| 10 | **Verification of Information Handled by MCP Servers** | **Medium risk.** The debugger processes runtime data from debugged applications — variable values, stack traces, expression evaluation results, CPU/memory profiles, and heap snapshots. This data is accurate (sourced directly from Chrome DevTools Protocol) but may contain sensitive information. All data stays local (no external transmission). The tool returns structured JSON with explicit `status`, `code`, and typed fields, reducing ambiguity. Input validation is performed via `zod` schemas on all tool inputs. |
| 11 | **Authentication Methods and Permissions** | **Available but disabled by default.** The `AuthManager` supports: (1) Token-based authentication with expiration (default 1 hour). (2) API key authentication with SHA-256 hashing. (3) Constant-time comparison to prevent timing attacks. (4) Token revocation and cleanup of expired tokens. The CLI has a `--no-auth` flag (authentication is on by default per CLI help text). The `MCP_AUTH_TOKEN` environment variable configures the auth token. **However:** In the actual `McpDebuggerServer` constructor, `AuthManager` is not instantiated or wired into the tool handlers — the security modules exist in the core library but are not actively enforced in the MCP server's request pipeline. **Recommendation:** This is a significant gap; auth should be wired in for TCP transport usage. |
| 12 | **Verification of AI Agent Execution Environment** | **Medium risk.** The debugger spawns child processes that inherit the parent's environment variables (`{ ...process.env }`), which may include cloud credentials, API keys, or other sensitive env vars. The `ResourceLimiter` enforces limits on concurrent sessions, breakpoints, memory, session duration, and watched variables — but these are not wired into the MCP server constructor either. The debugger operates within the user's local development environment. **Recommendation:** Sanitize environment variables before spawning debug targets; use the `env` configuration in launch configs to limit what's passed. |
| 13 | **Configure MCP Server Settings (Version Verification/Fixing)** | **Partially addressed.** The VS Code extension defaults to `npx -y @ai-capabilities-suite/mcp-debugger-server` which fetches the latest version — this means the version is NOT pinned by default. The `package.json` uses caret ranges (`^1.5.23`, `^1.0.16`) for internal dependencies. The server version is hardcoded in `cli.ts` (`VERSION = "1.5.23"`). **Risk:** `npx -y` will auto-install the latest published version without verification. **Recommendation:** Pin to a specific version in MCP configuration (e.g., `@ai-capabilities-suite/mcp-debugger-server@1.5.23`) and use `package-lock.json` or `yarn.lock` for reproducible installs. |
| 14 | **Verify MCP Server Connection Status During Prompt Input** | **Standard MCP behavior.** The server uses stdio transport by default, meaning it is started and managed by the IDE (VS Code/Kiro). Only one MCP server instance connects per configuration. The `.vscode/settings.json` and `.kiro/settings/mcp.json` explicitly list which MCP servers are active. The server does not auto-connect to other servers or forward prompts. **Recommendation:** Regularly review `.vscode/settings.json` `mcp.servers` and `.kiro/settings/mcp.json` to ensure only intended servers are configured. |
| 15 | **Listing and Managing Accounts/DBs/Containers, SQL Queries** | **Not applicable.** The MCP ACS Debugger does not access databases, containers, or execute SQL queries. It exclusively debugs Node.js/JavaScript processes via Chrome DevTools Protocol. It has no database drivers, ORM integrations, or container management capabilities. The only external resources it manages are spawned child processes (debug targets) and WebSocket connections to their inspector endpoints. |
| 16 | **Logging, Monitoring, and Log Query Capabilities** | **Comprehensive — available in core, partially wired.** The core library includes: (1) `StructuredLogger` — JSON-formatted logging with correlation IDs, session IDs, operation types, configurable log levels, and an in-memory buffer (1000 entries). (2) `AuditLogger` — tracks all operations with timestamps, session IDs, success/failure status, user IDs, and supports time-range queries, session filtering, and JSON export (10,000 entry capacity). (3) `MetricsCollector` and `PrometheusExporter` for metrics. (4) `HealthChecker` for health endpoints. The CLI supports `--log-level` and `MCP_LOG_LEVEL` env var. Console output goes to stderr. **Note:** Like auth, these modules exist in core but are not all actively wired into the MCP server's request handlers. |
| 17 | **Post-Approval Malicious Update Risk** | **Medium risk.** The default `npx -y` installation fetches the latest version from npm on each invocation, meaning a compromised npm publish could deliver malicious code without user review. The npm packages are published under the `@ai-capabilities-suite` scope, which requires npm org membership. The GitHub repo uses CI (GitHub Actions) for build/test/lint. **Risk factors:** (1) No package signing or provenance attestation observed. (2) Small team with potentially limited publish access controls. (3) No lock file enforcement in the default `npx` invocation. **Recommendation:** Pin versions, verify checksums, and monitor npm advisories for the packages. |
| 18 | **Outdated Dependencies** | **Low-Medium risk.** Runtime dependencies are minimal and mainstream: `@modelcontextprotocol/sdk@^1.23.0`, `zod@^4.1.13`, `tslib@^2.3.0`, `source-map@^0.7.4`, `ws@^8.18.3`. Dev dependencies include `jest@^29.7.0`, `typescript@^5.3.3`, `@swc/core@^1.15.3`. The project uses `yarn@4.12.0` with an install-state lockfile. Node.js >= 18.0.0 is required (18 is in maintenance LTS as of analysis date). **Recommendation:** Run `npm audit` / `yarn audit` regularly; consider upgrading the minimum Node.js requirement to 20 LTS. |
| 19 | **Environmental Damage due to Auto-Approval** | **High risk if auto-approved.** The `debugger_start` and `debugger_detect_hang` tools spawn real OS processes. The `debugger_inspect` tool evaluates arbitrary JavaScript in running processes. Auto-approving these tools would allow an AI agent to: (1) Execute arbitrary commands on the local machine. (2) Spawn processes that consume CPU/memory. (3) Evaluate expressions with side effects in debug targets. The `ResourceLimiter` (max sessions, max memory, max duration) exists but is not enforced in the current MCP server wiring. **Recommendation:** NEVER add these tools to `autoApprove` lists. Always require human review for `debugger_start`, `debugger_detect_hang`, and `debugger_inspect`. |
| 20 | **Intent/Objective Tampering** | **Low-Medium risk.** The tool descriptions in `package.json` and `mcp-server.ts` are clear and accurate — they describe debugging operations without hidden functionality. Input schemas are strictly defined via `zod` with required fields and typed parameters. Tool handlers perform the documented operation and return structured results. There is no mechanism for tool descriptions to be dynamically modified at runtime. **Risk factor:** The `debugger_inspect` tool evaluates arbitrary expressions, which could be used for purposes beyond inspection (e.g., modifying variables, calling functions with side effects). |
| 21 | **Human Operation** | **Medium risk.** Common human errors include: (1) Forgetting to stop debug sessions, leaving orphaned processes (mitigated by session timeouts and graceful shutdown). (2) Debugging production code with live credentials. (3) Using `--no-auth` in non-development environments. (4) Misconfiguring the MCP server path to point to a tampered binary. (5) Not reviewing AI agent tool calls before approval. The extension provides VS Code commands, keybindings, and a walkthrough to guide usage. **Recommendation:** Use supervised/non-autopilot mode when debugging; always review tool invocations. |
| 22 | **Lag Pull Attack** | **Low risk.** The debugger operates synchronously within debug sessions — each tool call operates on the current state of the paused process. Session IDs are validated on every call. There is no caching layer or delayed execution queue that could be exploited with stale data. Timeouts are enforced (default 30s for operations, 5s for process spawning). The `CircuitBreaker` pattern prevents repeated calls to failing operations. WebSocket connections to the inspector are direct and real-time. |
| 23 | **Cost-Related Information** | **Free.** MIT licensed, open source, no paid tiers, no usage fees, no API keys required for the tool itself. Published on npm (free), VS Code Marketplace (free), and Docker Hub (free). The provider is a 501(c)(3) nonprofit funded by GitHub Sponsors and donations. No subscription, no premium features, no rate-limited free tier. |

## Key Recommendations

1. **Never auto-approve** `debugger_start`, `debugger_detect_hang`, or `debugger_inspect` tools.
2. **Pin the version** in MCP configuration instead of using `npx -y` with latest.
3. **Enable DataMasker** when debugging applications that handle sensitive data.
4. The auth, rate-limiting, and resource-limiting modules exist in the core library but are **not wired into the MCP server request pipeline** — this is the most significant security gap.
5. Use supervised mode (not autopilot) when using this debugger with AI agents.
